Data Module Code

        
The address of every data module

        
Every data module in S1000D has a unique code — the DMC. It is not just a file name. It encodes who made it, what product it belongs to, what kind of information it contains, and more. Click each segment below to learn what it means.

Why the DMC matters

        

          

            
Unique identity

            
Two data modules can never share the same DMC. This makes every module globally identifiable across systems, organizations, and projects.

          

          

            
Searchable structure

            
Because each segment carries meaning, teams can filter and find modules by product, system, task type, or information category — without opening any file.

          

          

            
Machine-readable

            
The DMC is not just for humans. Tools and databases use it to sort, validate, link, and publish the right content automatically.

Information Codes

        
What kind of information is this?

        
The information code is the three-digit segment in the DMC that tells you — and any tool — what type of content a module contains. Knowing the most common codes is one of the fastest ways to start reading real S1000D projects.

      


      

        

          
Filter by category

          

        

      


      

        
      


      


      


      

        
How to read an information code

        

          

            
First digit — major category

            
The first digit groups codes into broad categories. 0xx is general or descriptive. 2xx covers fault and diagnostic content. 5xx covers maintenance tasks. 7xx covers adjustment and testing. 9xx is for crew and operator tasks.

          

          

            
Second and third digits — specific type

            
The remaining digits narrow the code to a specific information type within that category. 040 is a general description. 041 would be a more specific variation. Projects can define their own extensions within the ranges the standard allows.

          

          

            
Bicycle example

            
A module about how the braking system works as a whole would use 040 (Description). A module showing how to inspect brake pads would use 200-range (Fault isolation) or 280 (Inspection). A module walking through the adjustment steps would use 720 (Adjustment/Test).

BREX

        
BREX is a separate rule module

        
A BREX is not embedded inside the content module. It is its own S1000D data module that the content module refers to. Think of it as the project rule module that tells validation software what is allowed and what is not.

      


      

        
How to read the relationship

        

          

            
Content data module

            
This is the actual maintenance content, such as a bicycle chain inspection procedure.

            


<dataModule>
  <dmTitle>Inspect the bicycle chain</dmTitle>
  <content>...steps and warnings...</content>
  <brexDmRef>BREX-BIKE-0001</brexDmRef>
</dataModule>

            

          

          

            
BREX data module

            
This is a separate rule module. It does not contain the maintenance task. It contains validation rules for how project modules must be written.

            


<brexDataModule>
  <dmTitle>Bicycle project BREX</dmTitle>
  <brexDmRuleSpec>...project rules...</brexDmRuleSpec>
</brexDataModule>

            

          

        

        

          
Content module > refers to > BREX module
The content holds the maintenance information. The BREX holds the project rules.

Why BREX matters

        

          

            
Business rule > BREX rule

            
A project first decides a business rule in plain language, such as “all modules must use English.” BREX is the machine-readable version of that rule so software can check it consistently.

          

          

            
Validation tool

            
Software can validate every data module in the project against the BREX automatically. Any module that breaks a BREX rule is flagged before it reaches a technician. This prevents errors at the source.

          

          

            
Separate from the content

            
The procedure module and the BREX module are different modules with different jobs. One holds the maintenance content. The other holds the project rule set that the content must follow.
